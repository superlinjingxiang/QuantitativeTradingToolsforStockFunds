# AGENTS.md

## 项目使命

构建一个用于分析中国A股、场内ETF/LOF和场外公募基金的 Python 桌面应用。应用必须提供可搜索的市场数据、实时与历史图表、可解释的策略结果、事件驱动回测、风险控制和模拟交易功能。

系统目标是提高长期风险调整后的决策质量。系统**不得**宣称保证盈利，也不得在证据不足时强制给出交易建议。`ABSTAIN` / `不交易` 是合法且预期会出现的结果。

## 修改代码前必须阅读

采用渐进式上下文读取：只阅读当前任务相关的文档，但每次都应从以下文件开始：

1. `README.md`
2. `TASKS.md`
3. `docs/index.md` 中与任务相关的事实源文档
4. `docs/TRACEABILITY.md` 中的需求编号和验收编号

对于复杂功能、跨层重构、数据供应商接入、策略变更，或预计涉及多个模块的工作，必须在 `docs/exec-plans/active/` 下创建或更新执行计划（ExecPlan），并遵循 `PLANS.md`。

## 不可妥协的领域约束

- 国际通用金融与统计理论用于提出假设；中国市场规则用于约束交易执行。
- 不得把单一K线形态或单个技术指标作为高置信度交易的充分证据。
- 普通A股、场内基金和场外基金不得共用一套笼统的执行规则；必须按交易所、资产类型、具体证券和规则生效日期解析。
- 回测必须使用时点数据、按时间顺序运行，并避免已知的未来数据泄漏。
- T日收盘后产生的信号，不得使用信号产生时尚不可获得的价格成交。
- 必须建模佣金、税费、滑点、买卖价差、流动性、部分成交、停牌、涨跌停、公司行为和可卖数量。
- 场外基金的盘中估算净值不得进入正式净值回测。
- 数据陈旧、缺失、不一致、越权、分布外或未授权时，必须阻止生成新的正式交易信号。
- 每个信号必须包含数据时间、数据快照、策略/模型/规则版本、有效期、解释，以及适用时的不交易原因。
- MVP只提供分析和模拟交易。未经单独批准的规格与合规审查，不得加入真实订单提交能力。

## 架构边界

- UI代码不得直接调用行情数据供应商。
- 策略不得实现交易所规则或成交假设。
- 规则解析属于规则引擎。
- 供应商特有字段的标准化属于数据适配器。
- 风险决策发生在原始策略输出之后、订单创建之前。
- GUI中的网络请求、策略计算和回测计算不得阻塞 Qt 主线程。
- 共享领域逻辑必须能够脱离GUI独立运行。

依赖方向详见 `docs/architecture/ARCHITECTURE.md`。

## 工程约定

- 实现语言为 Python；创建 `pyproject.toml` 后，以其中固定的 Python 版本为准。
- 公共API和领域模型必须使用类型注解。
- 指标、因子、费用模型和规则判断优先使用小型、确定性的函数。
- 外部I/O必须隐藏在协议/接口之后，并使用假实现进行测试。
- 内部时间戳尽量使用带时区的UTC时间；交易日和交易时段必须显式保留交易所本地语义。
- 不得隐藏降级或兜底行为；数据降级和模型降级必须通过有类型的结果暴露。
- 新增生产依赖前，必须在 `docs/DECISIONS.md` 记录用途、替代方案与取舍。
- 窄范围修改足以满足需求时，避免大范围重写。

## 必须执行的验证

在宣布任务完成前：

- 为变更行为新增或更新测试。
- 先运行最小相关测试集，再运行仓库规定的完整检查。
- 配置完成后，运行格式化、代码检查和类型检查。
- 核对 `docs/quality/ACCEPTANCE_CRITERIA.md` 中受影响的验收条件。
- 当行为或范围发生变化时，更新 `docs/TRACEABILITY.md`、`TASKS.md`、契约和对应事实源文档。
- 审查差异，重点检查未来数据泄漏、陈旧数据处理、成交真实性、UI竞态和结果可复现性。

只有实现、测试、文档、错误状态、可观测性和可复现性证据全部完成，变更才算完成。详见 `docs/quality/DEFINITION_OF_DONE.md`。
