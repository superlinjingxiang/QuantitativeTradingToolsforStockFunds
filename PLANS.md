# Codex 执行计划规范

执行计划（ExecPlan）是一份可持续更新的复杂任务实施规格。它必须让一个不了解此前对话的工程师或新的Codex会话，仅依靠仓库与计划即可继续工作。

## 何时必须创建执行计划

以下工作需要创建或更新执行计划：

- 跨越一个以上架构层的功能；
- 新数据供应商接入或交易所规则实现；
- 策略、模型、回测、组合或风险行为变更；
- 涉及异步状态、取消或多个面板联动的GUI工作；
- 数据迁移、重大重构或存在实质性未知项的任务；
- 预计包含多个里程碑或需要较多调研的工作。

小型文档修复和独立的确定性缺陷，可以在任务回复中使用简短计划。

## 必须具备的属性

- **自包含**：说明仓库背景、相关路径、术语和假设。
- **面向结果**：描述可观察行为，而不是只列出要修改的文件。
- **持续更新**：每个里程碑后更新进度、决策、发现和剩余工作。
- **可验证**：列出准确的命令和预期证据。
- **可恢复**：记录足够状态，使新会话无需聊天历史也能继续。
- **明确不确定性**：可行性未知时，使用概念验证里程碑。

## 必须包含的章节

使用以下结构：

```markdown
# <面向结果的计划标题>

## 目的和用户可见结果
## 背景与仓库定位
## 范围
### 范围内
### 范围外
## 需求编号与验收编号
## 进度
- [ ] YYYY-MM-DD HH:MMZ — 里程碑
## 意外情况与发现
## 决策日志
- YYYY-MM-DD — 决策、理由、备选方案
## 架构与接口
## 里程碑
### 里程碑1 — ...
### 里程碑2 — ...
## 具体实施步骤
## 验证与验收
## 可复现性、幂等性与恢复
## 风险与缓解措施
## 产物与备注
## 结果与复盘
```

## 执行行为要求

执行计划时：

1. 不要反复询问泛化的“下一步”，应按计划继续到下一个里程碑。
2. 每个停止点都要保持计划为最新状态。
3. 在决策日志中记录范围变化和取舍。
4. 每个里程碑后执行相关验证，不得全部拖到最后。
5. 没有可观察证据，不得宣称成功。
6. 完成后将计划移到 `docs/exec-plans/completed/`，并从 `TASKS.md` 或 `docs/TRACEABILITY.md` 链接。
