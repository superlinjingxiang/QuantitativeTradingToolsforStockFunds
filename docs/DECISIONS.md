### ADR-004——Python 3.12、uv与首轮PySide6依赖
- 日期：2026-06-28
- 状态：已接受
- 背景：TASK-001需要可复现安装、锁文件、测试/类型/代码检查命令，并为后续PySide6桌面GUI保留一致运行时。
- 决策：项目固定Python 3.12，使用 `uv` 和 `pyproject.toml` 管理依赖与锁文件；首轮将 `PySide6` 作为生产依赖加入，但仅做导入冒烟测试，GUI外壳留到TASK-008。
- 已考虑的替代方案：Poetry、pip-tools、延后加入PySide6。Poetry和pip-tools不作为首选；延后PySide6会降低TASK-008前的环境确定性。
- 影响：CI和本地开发统一使用 `uv sync --all-extras --dev`；`uv.lock` 成为可复现安装证据；基础测试覆盖PySide6 `QtCore` 导入。
- 受影响的需求：TASK-001、NFR-08。
- 所需测试/迁移：`ruff format --check`、`ruff check`、`mypy src tests`、`pytest`、`python -m china_quant_platform --version`。

### ADR-005——使用Pydantic实现领域契约模型
- 日期：2026-06-28
- 状态：已接受
- 背景：TASK-002要求标准领域模型、Schema双向转换、时区时间戳校验、概率归一化和可交易报告不变量。
- 决策：使用Pydantic v2实现不可变领域模型，并将 `pydantic` 作为显式生产依赖；所有面向契约的模型提供 `to_contract_dict()` 和 `from_contract_dict()`。
- 已考虑的替代方案：标准库 `dataclasses` 加手写校验、直接使用原始字典。手写校验会重复Schema规则，原始字典不满足有类型领域模型要求。
- 影响：领域层获得统一运行时校验；`DataHealth`、`AnalysisReport`、`BacktestConfig` 和 `SecurityRule` 可与 JSON Schema 往返；后续供应商和GUI必须使用这些模型而不是原始字典。
- 受影响的需求：TASK-002、FR-020、AC-06、AC-10、AC-11。
- 所需测试/迁移：Schema往返测试、非法概率测试、naive时间戳测试、可交易报告来源信息测试、正式/估算基金净值隔离测试。

### ADR-006——使用确定性假供应商验证数据协议
- 日期：2026-06-28
- 状态：已接受
- 背景：TASK-003要求供应商协议不绑定具体SDK，并能在没有真实行情凭据时验证历史、实时、取消和限流行为。
- 决策：在 `data` 层定义 `MarketDataProvider` 协议、请求模型和能力声明；实现 `DeterministicFakeMarketDataProvider` 作为无外部I/O的契约测试供应商。

- 影响：后续适配真实供应商必须实现同一协议；领域层继续不依赖 `data` 层或任何供应商SDK；缺能力通过类型化 `DataUnavailable` 暴露。
- 受影响的需求：TASK-003、NFR-06、NFR-08。
- 所需测试/迁移：供应商协议测试、确定性fixture测试、实时订阅测试、取消测试、限流测试和领域层依赖边界测试。

### ADR-007——首版证券主数据采用内存索引
- 日期：2026-06-29
- 状态：已接受
- 背景：TASK-004需要本地候选搜索、别名匹配、时点状态和最近搜索；SQLite持久化与GUI切换事务在后续任务中实现。

- 影响：搜索路径无外部I/O，适合GUI前的单元/集成测试；后续持久化层可用同一服务接口加载记录。
- 受影响的需求：TASK-004、FR-001、AC-01、NFR-01。
- 所需测试/迁移：T-10搜索测试、时点状态测试、最近搜索测试、2000证券夹具P95搜索性能测试。

### ADR-008——历史K线缓存使用pyarrow写入Parquet分区
- 日期：2026-06-29
- 状态：已接受
- 背景：TASK-005要求历史行情按证券、周期和年份保存为Parquet，并且请求前计算缺失区间，只补齐缺口。
- 决策：将 `pyarrow` 作为生产依赖；实现 `HistoricalBarCache` 以单分区文件保存标准 `Bar` 契约行，并由 `MarketDataGateway` 协调缓存读取、缺口请求、幂等追加和实时订阅状态。
- 已考虑的替代方案：CSV/JSON临时缓存、DuckDB优先、Polars优先。CSV/JSON不满足Parquet交付要求；DuckDB跨分区分析留给后续存储查询；Polars对当前单文件读写需求不是必需。
- 影响：历史K线缓存具备可复现的本地列式存储；同一证券、周期、复权和时间戳唯一；实时Quote状态可转换为阻断信号的 `DataHealth`。
- 受影响的需求：TASK-005、FR-003、FR-004、AC-06、NFR-02、NFR-04。
- 所需测试/迁移：Parquet分区读写、重复时间戳、损坏OHLC、缓存幂等追加、增量补缺、陈旧Quote、订阅取消和断线重连测试。

### ADR-009——数据质量门禁输出结构化阻断报告
- 日期：2026-06-29
- 状态：已接受
- 背景：TASK-006要求新鲜度、完整性、一致性和授权检查，并保证被阻断数据不得生成最终可交易信号。
- 决策：实现 `DataQualityService`，输出结构化 `DataQualityIssue` 与 `DataQualityReport`，再由报告生成标准 `DataHealth`；阻断状态通过 `assert_signal_allowed()` 映射为类型化 `DataStale`、`DataInvalid` 或 `UnauthorizedData`。
- 已考虑的替代方案：仅在供应商适配器中抛异常、把检查散落在策略层。适配器抛异常难以汇总多项质量问题；策略层分散检查会破坏统一阻断语义。
- 影响：上层策略和GUI可以统一读取健康状态、问题列表和阻断原因；跨源Quote对账可配置容差；历史完整性复用缓存时间窗逻辑。
- 受影响的需求：TASK-006、FR-003、FR-004、FR-020、AC-06、AC-10、C-005。
- 所需测试/迁移：T-08、T-16，重复K线、非法OHLC、缺失K线、陈旧Quote、缺失字段、授权、跨源对账和可交易报告阻断测试。

### ADR-010——首版市场规则使用内存仓库按生效日期解析
- 日期：2026-06-29
- 状态：已接受
- 背景：TASK-007要求规则不得硬编码为永久常量，缺失规则时必须关闭式失败，并覆盖中国市场的T+、涨跌停、停牌、费用和基金语义。
- 决策：实现 `InMemoryRuleRepository` 与 `MarketRuleEngine`；规则仍使用标准 `SecurityRule` 契约，交易细节作为审核规则的扩展字段解释；解析时按证券专属优先、交易所、资产类型、生效日期和审核状态选择规则。
- 已考虑的替代方案：直接用证券代码前缀推断规则、现在引入数据库规则表。前缀推断违反规格；数据库持久化留给后续存储/迁移任务。
- 影响：回测和信号层可通过同一规则入口校验订单、价格带、可卖数量和费用；缺失或冲突规则以 `RuleMissing` 关闭式失败。
- 受影响的需求：TASK-007、C-002、C-004、AC-07、AC-08、AC-11。
- 所需测试/迁移：T-02至T-06、T-15，有效期边界、证券专属优先、缺规则失败、T+1、涨跌停流动性、停牌、费用、基金净值语义和披露前不可见测试。

### ADR-011——GUI外壳采用ViewModel状态快照
- 日期：2026-06-29
- 状态：已接受
- 背景：TASK-008要求PySide6主窗口、状态模型、导航页签、后台任务抽象和数据健康横幅，同时Qt主线程必须保持响应。
- 决策：实现 `ApplicationViewModel` 持有不可变 `AppUiState` 快照，`MainWindow` 只订阅状态并渲染；首版 `CancellableQtTask` 使用Qt定时器抽象演示非阻塞任务、取消和类型化错误。

- 影响：GUI层获得可测试状态机和健康横幅；后续搜索、订阅和图表更新必须通过ViewModel或应用服务进入状态。
- 受影响的需求：TASK-008、AC-02、AC-06、NFR-03、NFR-06。
- 所需测试/迁移：pytest-qt状态渲染、非阻塞取消和类型化错误状态测试。

### ADR-012——证券搜索与切换事务归属ViewModel
- 日期：2026-06-29
- 状态：已接受
- 背景：TASK-009要求搜索控件、键盘操作、`selection_generation` 和旧订阅取消，并且旧证券任务结果不得覆盖新证券状态。
- 决策：`MainWindow` 只负责300ms防抖、候选列表和键盘/鼠标事件；`ApplicationViewModel` 调用 `SecurityMasterService`、记录最近访问、递增 `selection_generation`、取消旧任务并丢弃旧generation结果。
- 已考虑的替代方案：在Widget中直接搜索和选择、使用全局状态单例。Widget直连业务难以测试竞态；全局状态会破坏多窗口/测试隔离。
- 影响：后续历史加载、实时订阅、图表刷新和策略计算必须携带选择代次，避免旧代结果覆盖当前上下文。
- 受影响的需求：TASK-009、FR-001、FR-002、AC-01、AC-02、NFR-06。
- 所需测试/迁移：T-07/T-10，GUI防抖搜索、键盘候选选择、选择代次、旧任务取消和旧generation结果丢弃测试。

### ADR-014——首版指标与因子使用纯Python确定性计算
- 日期：2026-06-29
- 状态：已接受
- 背景：TASK-011要求指标是纯函数或可缓存的确定性计算，必须处理预热期、缺失值和未来函数风险；当前仓库尚未引入pandas/numpy作为运行时依赖。
- 决策：在 `indicators` 中用标准库实现SMA/EMA/MACD/RSI/ATR/布林带/VWAP、收益、波动、下行波动、回撤和相对强弱；在 `factors` 中实现版本化 `FactorMetadata`、`FactorRegistry` 和默认点时因子。缓存键使用排序JSON与SHA256生成。
- 已考虑的替代方案：立即引入pandas或TA-Lib、把指标直接写在策略层。pandas/TA-Lib会增加依赖和安装面；策略层内嵌指标会削弱复用和审计。
- 影响：指标和因子不依赖GUI、供应商或真实数据库；后续策略/回测可以复用同一组点时计算与缓存键。
- 受影响的需求：TASK-011、FR-012、AC-10、NFR-08。
- 所需测试/迁移：预热期、缺失值、未来值隔离、因子元数据、重复注册、默认因子计算和缓存键排序无关性测试。

### ADR-015——策略层只输出原始信号与解释
- 日期：2026-06-29
- 状态：已接受
- 背景：TASK-012要求策略协议、原始信号、适用/失效条件和解释，同时明确策略不得直接产生最终订单或绕过风险与规则层。
- 决策：在 `strategies` 中定义 `Strategy` 协议、`StrategyContext`、`WarmupSpec`、`StrategyMetadata`、`RawSignal`、`Explanation` 和 `evaluate_strategy()`；`RawSignal` 使用独立 `RawSignalIntent`，强制 `requires_data_quality_gate`、`requires_rule_gate` 和 `requires_risk_gate` 为真，并禁止订单/最终信号字段。
- 已考虑的替代方案：让策略直接输出 `FinalSignal` 或目标仓位、把解释作为自由文本。直接输出最终决策会绕过规则/风险流水线；自由文本解释难以审计和测试。
- 影响：后续回测和风控层可以把策略输出作为受控输入，继续在数据、规则、风险和执行层生成订单意图、成交和最终信号。
- 受影响的需求：TASK-012、FR-006、FR-018、AC-04、AC-10。
- 所需测试/迁移：RawSignal越权字段拒绝、门禁不可关闭、预热不足阻断、策略/证券/快照身份一致性和解释一致性测试。

### ADR-016——回测内核先固定事件时序与规则拒绝语义
- 日期：2026-06-29
- 状态：已接受
- 背景：TASK-013要求事件循环、交易时钟、订单/成交/拒绝/部分成交、取消和审计；成本、滑点、流动性容量和公司行为将在TASK-014细化。
- 决策：实现 `BacktestEventLoop` 以时间、事件优先级和序列排序；`BacktestEngine` 只在市场事件后评估策略，订单必须晚于信号时间，并通过 `MarketRuleEngine` 校验后生成 `FILL`、`PARTIAL_FILL` 或 `REJECT` 事件。运行结果用事件快照SHA256作为checksum。
- 已考虑的替代方案：在TASK-013直接实现完整投资组合/成本/滑点、或用线性for循环替代事件队列。完整执行模型会扩大任务边界；线性循环难以处理跨日订单、取消和后续异步事件。
- 影响：TASK-014可在同一执行模拟器位置接入费用、滑点、流动性和公司行为；TASK-016可复用事件checksum和审计记录做回归。
- 受影响的需求：TASK-013、FR-013、AC-07、AC-09、NFR-05、NFR-08。
- 所需测试/迁移：事件优先级、策略时序、订单时间防泄漏、规则拒绝、部分成交、取消和重复运行checksum一致测试。

### ADR-017——执行模型以可组合小模型接入回测模拟器
- 日期：2026-06-29
- 状态：已接受
- 背景：TASK-014要求佣金、税费、滑点、价差、流动性、涨跌停、延迟和公司行为；其中涨跌停/停牌/手数仍应由市场规则引擎统一负责。
- 决策：新增 `backtest.execution`，提供 `RuleBasedCostModel`、`FixedBpsSlippageModel`、`FixedSpreadModel`、`ParticipationRateLiquidityModel`、`FixedLatencyModel` 和 `CorporateActionProcessor`；`DeterministicExecutionSimulator` 组合这些模型并继续调用 `MarketRuleEngine` 做规则校验。
- 已考虑的替代方案：把所有执行逻辑写进 `BacktestEngine`、或现在引入真实券商/盘口模型。集中写入引擎会难以替换和测试；真实盘口模型需要数据契约和供应商接入，超出当前任务。
- 影响：后续可在不改策略接口的情况下替换更真实的滑点、盘口和容量模型；组合对账可复用公司行为处理器。
- 受影响的需求：TASK-014、AC-08、AC-11、T-02至T-06、T-15、T-18。
- 所需测试/迁移：费用/税费、滑点/价差、参与率部分成交、延迟、涨停无对手方流动性拒绝、分红和拆股测试。

### ADR-018——组合与风险引擎分离记账和门禁
- 日期：2026-06-29

### ADR-031——同花顺优先但保留多源行情兜底
- 日期：2026-06-30
- 状态：已接受
- 背景：东方财富公开接口在搜索、quote和K线请求上会间歇性断开连接，用户日常使用同花顺App，希望默认优先采用同花顺数据源。
- 决策：新增同花顺 iFinD/QuantAPI Provider，并通过 `MultiSourceMarketDataProvider` 按“同花顺 -> 东方财富/Yahoo兜底”顺序路由。只有配置 `CQP_THS_IFIND_REFRESH_TOKEN` 时才启用同花顺，避免把付费/授权数据源变成硬依赖；真实 token 只能来自环境变量或本机忽略的 `.env`，不得提交。

### ADR-037——图表最大利润回测层只作为历史复核工具
- 日期：2026-07-01
- 状态：已接受
- 背景：用户希望在图表顶部增加一个回测按钮，按当前周期、策略期限和交易次数在历史区间中找到利润最大化路径，并直接在折线图上展示何时买入、卖出。该能力有助于理解“如果事后知道走势，最佳买卖点在哪里”，但天然使用完整历史价格，不能混同为未来预测算法。
- 决策：在现有 PySide 图表上增加“回测曲线/正常显示”双态按钮，按钮本身保持 checked/active 状态同步；按当前策略期限截取图表窗口，以动态规划求解最多 N 笔完整买卖后的最大资金曲线；交易次数可以少于上限但不得超过上限。进入回测层前保存当前图表与回测面板快照，结果写入 `BacktestPanelState`，并通过 `ChartSignalMarkerState` 在价格折线上绘制 B/S 标记和悬停说明；再次点击时恢复点击前的正常图表、叠加层和回测面板。该层状态标记为 `OPTIMIZED`，文案明确说明属于历史最大利润路径和上帝视角，不提升真实执行候选等级。
- 已考虑的替代方案：把“重新回测”按钮复用为最大利润路径，或把最大利润结果直接当作策略信号。复用底部按钮会让用户不清楚是否已开启图表层；把结果当作策略信号会产生未来函数和收益承诺风险。
- 影响：用户可以直观看到当前窗口内的最佳历史买卖点和交易流水；盈利验证策略、DecisionHub、模拟盘和真实API边界保持不变。后续若接入Electron，前端应复用同一后端回测层输出，而不是在前端重新计算。
- 受影响的需求：TASK-034、FR-013、FR-014、FR-021、FR-022、AC-07、AC-09、EPV-001、EPV-002。
- 所需测试/迁移：“回测曲线/正常显示”双态按钮GUI测试、最大利润交易次数上限测试、B/S 标记渲染测试、正常显示快照恢复测试、回测面板交易流水测试和全量ruff/mypy/pytest回归。

### ADR-039——盈利验证策略加入量能和流动性确认
- 日期：2026-07-02
- 状态：已接受
- 背景：用户提供的金融指标参考强调不要只看单一价格指标，应结合趋势、动量、波动、成交量/流动性和组合风险。当前 `profit_validation` 第一版已经具备动量、趋势、波动和回撤过滤，但入场缺少量能确认，容易把低参与度的价格上涨误判为可交易趋势。

- 影响：策略会减少弱量能上涨中的入场次数，提高研究信号的可解释性；该优化不改变真实交易门禁，仍需样本外、成本、回撤和模拟盘证据共同确认。
- 受影响的需求：TASK-036、FR-013至FR-015、FR-021、FR-022、EPV-001至EPV-006。
- 所需测试/迁移：量能确认过滤单元测试、全量盈利验证回归、GUI回测面板说明检查和后续真实ETF样本诊断。

### ADR-042——Redis优先、内存降级的短期API缓存
- 日期：2026-07-11
- 状态：已接受
- 背景：行情供应商公开接口间歇失败，重复打开同一标的会重复请求，用户看到空白或必须手动刷新；已有Parquet缓存适合历史K线，但不适合短期API状态共享。
- 决策：新增统一缓存抽象，Redis可用时缓存健康、搜索、分析和荐股结果；Redis不可用时自动降级到进程内缓存。历史K线继续以Parquet为持久缓存。缓存命中、未命中和陈旧回退都写入响应元数据；陈旧数据必须标记并阻断新的正式信号。
- 已考虑的替代方案：强制用户安装Redis，或只使用浏览器内存缓存。强制Redis会破坏单击运行，浏览器缓存无法覆盖Electron后端和多进程场景。
- 影响：重复加载更快，数据源失败时保留上次可见结果；缓存不会伪造实时行情、不会绕过数据质量门禁。
- 所需测试/迁移：内存缓存命中、TTL、陈旧回退、Redis不可用降级、数据健康阻断和自动刷新失败保留上次结果测试。

### ADR-043——用风险调整收益和滚动一致性约束盈利验证通过
- 日期：2026-07-12
- 状态：已接受
- 背景：仅凭最终留出区间的正收益或正超额，可能把单一行情阶段的偶然结果误判为稳定策略；部分低回撤策略也会因总收益略低于基准被简单否决。
- 决策：盈利验证 V2 同时记录 Sharpe、Calmar、基准最大回撤和滚动前推折。PASS 必须满足正收益、回撤上限和足够交易数，并满足正超额，或满足 Sharpe 至少 0.75 且相对基准回撤改善至少 20%。最终留出通过但滚动正收益折低于 55%、折中位收益不为正或有效折少于 5 时降级为 WATCH。当前 BUY 还必须通过预测校准门禁。
- 已考虑的替代方案：只优化最终样本外收益，或为每类标的手工选择最优参数。两者都会增加数据窥探和过拟合风险。
- 影响：界面更少出现缺乏跨窗口证据的 BUY；风险控制更可解释，但不会把风险调整指标包装成收益保证。
- 所需测试/迁移：阈值无泄漏、滚动前推一致性、风险调整比较、预测门禁、十 ETF 和混合十标的真实复核。

### ADR-044——成本压力必须复用交易路径，无交易折必须进入一致性分母
- 日期：2026-07-13
- 状态：已接受
- 背景：旧实现以 `cost_drag` 字段存在判断成本压力，可能把基础成本结果误标为压力通过；滚动前推只统计有交易折，也可能高估长期空仓策略的正收益比例。真实多资产复核还发现公开行情源可能返回被截断的少量历史数据但不报错。
- 决策：V3 用同一阈值和同一交易路径把往返成本从 15bp 提高到 45bp，不重新调参；压力收益非正或回撤越界时降级。滚动前推全部折进入正收益和中位数统计，无交易折按 0 收益计入，并单独要求至少 50% 参与率。长周期行情明显截断时尝试备用源，实验室明确报告历史条数不足。
- 已考虑的替代方案：压力场景重新挑阈值、继续排除无交易折、把短历史直接交给策略。三者都会分别引入二次过拟合、选择性统计或把数据问题伪装成算法结果。
- 影响：混合十标的 PASS 从旧口径 3/10 降为 2/10，界面会显示真实成本压力和滚动参与率；结果更保守，但证据语义更可靠。实验性市场状态和单日追涨过滤未通过跨窗口稳健性检查，保持默认禁用。
- 所需测试/迁移：成本压力固定路径测试、成本证据 MISSING/FAIL 门禁测试、无交易折滚动统计测试、K线截断备用源测试、空列表切源测试和两组真实十标的复跑。

### ADR-045——短线策略按目标波动控制风险暴露并固化数据快照
- 日期：2026-07-13
- 状态：已接受
- 背景：V3 固定满仓会让高波动股票对净值和回撤贡献过大；同时公开行情源可能回补或修正历史数据，仅有结果校验和不足以解释复跑差异。

### ADR-046——A股短线入场采用资产专属反追涨约束
- 日期：2026-07-13
- 状态：已接受
- 背景：V4 在十 ETF 上保留 2/10 PASS，但五只 A 股为 0/5 PASS。继续扩大到消费、银行、保险、医药、制造、券商、公用事业、金融科技、电子和资源材料十类 A 股后，发现部分入场发生在单日急涨或中短期动量过热之后。若直接按最终留出收益挑阈值，会产生数据窥探。

- 所需测试/迁移：A 股/ETF 分流单元测试、十 A 股行业桶与去重测试、开发折选择和最终留出隔离检查、十 A 股/十 ETF/混合十标的真实复跑、四个决策模块版本与支持依据展示测试，以及后续模拟盘可交易性验证。

### ADR-047——短线回测使用次日开盘退出并执行A股可交易性约束
- 日期：2026-07-13
- 状态：已接受
- 背景：V5 入场信号已在次日开盘成交，但持有期、评分和趋势退出仍可能使用当日收盘数据并在同一收盘价成交，形成不可执行的时间优势；自定义盈利验证器也没有应用普通 A 股 T+1、停牌和一字涨跌停阻断。
- 决策：V6 将所有收盘后才能确认的退出信号推迟到下一交易日开盘；止损按“隔夜越过则开盘、盘中触发则止损价”建模；A 股个股禁止买入当日卖出，并在停牌、零成交量、一字涨停买入和一字跌停卖出场景阻断成交。结果必须记录次日开盘退出、同日退出、T+1 延迟、拒绝买入和延迟卖出次数，并在界面决策证据中展示。
- 已考虑的替代方案：继续按信号日收盘成交、仅在文档声明偏差、或直接接入复杂逐笔撮合。前两者不能消除时间泄漏；逐笔撮合超出当前公开日线数据能力。V6 先采用可复现的保守日线执行模型，并明确保留盘口排队、容量和 ETF 交易制度缺口。
- 影响：十 A 股最终留出平均收益由 V5 的 5.71% 降为 -0.14%，中位收益由 8.21% 降为 -2.44%，仍为 0/10 PASS。收益下降是消除理想成交偏差后的预期结果，证明当前短线策略尚不具备跨 A 股稳定盈利证据；执行边界继续保持 `RESEARCH_ONLY`。
- 受影响的需求：TASK-041、US-06、US-09、FR-006至FR-009、FR-013至FR-015、FR-020至FR-022、EPV-001至EPV-006。
- 所需测试/迁移：次日开盘退出、A 股 T+1、一字涨停拒买、一字跌停延迟卖出、ETF 不误套规则、执行统计序列化、右侧决策证据展示、三组十标的真实复跑，以及后续 ETF T+0/T+1 分类和模拟盘成交偏差验证。

### ADR-048——A股短线新仓使用按时点市场环境门槛并失败关闭
- 日期：2026-07-14
- 状态：已接受
- 背景：V6 修复成交时序后，正式十 A 股平均收益降为负值。开发折研究显示，在市场短中期趋势同时为正时限制个股新仓可以减少部分逆势交易，但单一最终样本无法证明稳健性；市场代理数据缺失也不能被静默视为允许交易。
- 决策：V7 对 A 股个股新开仓增加沪深300ETF `SSE:510300` 市场环境门槛，只使用信号日及以前数据计算 21 日和 63 日动量，两者都不低于 0% 才允许入场。代理历史不足或异常时状态为 `MISSING` 并阻断新仓。该门槛不阻止已有仓位退出，也不作用于 ETF。日线跟踪止损同时改为先用前一日已确认峰值判断当日止损，再更新峰值，消除当日高点前视。
- 已考虑的替代方案：缺失代理时放行、用当期最终状态覆盖历史信号、或对 ETF 统一套用 A 股门槛。缺失放行会把数据问题变成交易许可；最终状态覆盖历史会引入未来信息；ETF 跨资产驱动不同，统一门槛会混淆策略路径。
- 影响：正式十 A 股平均收益改善到 1.05%、平均回撤改善到 -10.36%，但正式、确认、影子三套池全部为 0/10 PASS，确认池平均亏损 -7.93%。因此 V7 只证明风险过滤候选值得继续研究，不证明稳定赚钱，执行状态继续保持 `RESEARCH_ONLY`。
- 受影响的需求：TASK-042、US-03、US-05、US-06、US-09、FR-006至FR-009、FR-013至FR-015、FR-020至FR-022、EPV-001至EPV-006。
- 所需测试/迁移：市场门槛按时点无泄漏、上升/下降/缺失代理、ETF不受影响、三套股票池互斥、FastAPI和GUI门禁展示、五组真实复跑，以及后续行业状态、容量和模拟盘成交偏差验证。

### ADR-049——A股个股未通过盈利验证时阻断新增仓位，ETF按独立证据判断
- 日期：2026-07-14
- 状态：已接受
- 背景：V7及多个候选在四组固定A股池和一个额外跨行业十股池中均未形成通用PASS；相同V7路径在十ETF上仍有2/10 PASS和正的聚合样本外收益。继续用统一WATCH语义会掩盖资产类别证据差异，也可能让账户模块把证据不足误解为可加仓。
- 决策：短线策略公式继续保持 `profit-validation-short-v7`。决策规则升级为 `rules-profit-validation-short-asset-gate-v1`：A股个股若本标的样本外状态未PASS，非卖出风险场景输出ABSTAIN并将目标新仓归零；若预测明确下跌，仍允许SELL/REDUCE用于已有仓位风险管理。ETF不套用这条个股门禁，继续按自身样本外、滚动前推、成本和校准证据输出。手动账户只消费同一个最终信号和仓位上限。

- 所需测试/迁移：A股WATCH证据转ABSTAIN、ETF保持WATCH、卖出/减仓优先、账户空仓不新增、Vue多项证据展示、十股票和十ETF复跑，以及后续横截面选股与模拟盘验证。

### ADR-050——ETF组合轮动采用正绝对动量、目标波动和最少滚动折门槛
- 日期：2026-07-17

### ADR-051——预测区间校准使用不重叠评估时点和持有期隔离
- 日期：2026-07-17
- 状态：已接受
- 背景：21日预测旧版逐日验证，连续标签高度重叠；历史评估时还可能使用当时尚未完整兑现的近端未来收益标签。固定80个表面样本因此会夸大有效样本量，并产生时点泄漏风险。
- 决策：`forecast.similar_regime_interval.v2` 按预测期限设置评估步长，21日预测只统计相隔21日的独立时点；每个历史时点的训练集隔离完整持有期，只允许使用当时已经兑现的标签。方向评分改为上涨/横盘/下跌三分类Brier，并按期限要求最少独立样本。校准结构、样本数、覆盖率、下破率和Brier进入 `AnalysisReport` 及 DecisionHub 门禁。缓存命名空间升级为 `v2`，避免旧证据继续命中。
- 已考虑的替代方案：继续保留逐日验证但用统计修正，或只在文档中声明样本相关性。前者仍难消除训练标签的时点泄漏，后者不能阻止不可靠结果提升执行状态。
- 影响：第一组十标的平均区间覆盖由旧口径88.88%降为去重后的82.67%，结果更保守；两组独立十标的仍达到当前HIGH聚合门槛，但该结论只证明预测校准质量，不证明策略盈利。账户评估继续消费同一最终信号和仓位上限，不产生第二套策略。
- 受影响的需求：TASK-046、US-04、US-05、US-09、FR-007、FR-020至FR-022、EPV-003、EPV-005。
- 所需测试/迁移：隔离区未来标签扰动不变测试、期限样本门槛、契约往返、DecisionHub PASS/MISSING/FAIL、两组十标的真实复跑和后续跨市场状态/模拟盘校准。

### ADR-052——ETF组合证据作为单标的附加门禁并与账户共用最终决策链
- 日期：2026-07-17
- 状态：已接受（研究门禁，未开放执行）
- 背景：V9 ETF组合轮动已有固定池、次日开盘、成本压力和时间留出证据，但旧界面只能看到单标的策略。若直接把组合排名改写成单标的买入信号，会混淆两种模型；若账户只使用单标的原始信号，则可能绕过DecisionHub证据门禁。
- 决策：`AnalysisReport`增加可选组合证据；组合快照必须复用回测的同一路径。DecisionHub把组合状态作为附加门禁：WATCH、陈旧或未入选不得许可新增仓位。账户评估同时接收原始策略与最终门禁；空仓服从最终门禁，已有仓位只在策略明确减仓/卖出或超过研究仓位上限时降低，不能把证据不足产生的WATCH解释为强制清仓。缓存Schema升级为v3。
- 已考虑的替代方案：把V9直接替换单标的策略、只在界面显示组合文本、或让账户继续读取原始信号。三者分别会混淆策略语义、缺少机器可验证门禁、或造成账户绕过证据链。
- 影响：固定十ETF可在四个模块看到组合状态、排名和研究仓位；V9留出段仍为WATCH，因此不能新增仓位或升级PAPER_READY。非固定池标的不受组合附加门禁影响。
- 受影响的需求：TASK-027、TASK-045、TASK-047、US-09、FR-008、FR-009、FR-015、FR-020至FR-022、EPV-001至EPV-006。
- 所需测试/迁移：组合证据契约往返、当前配置无前视、WATCH/未入选门禁、账户空仓与已有持仓语义、缓存旧值标识、固定十ETF真实复跑和后续模拟盘/容量验证。

### ADR-053——ETF容量使用按时点ADV和平方根冲击模型进入同一决策链
- 日期：2026-07-20
- 状态：已接受（第一阶研究门禁，非盘口成交证明）
- 背景：V9 ETF组合已有次日开盘、基础/压力成本和时间留出证据，也已经接入DecisionHub与手动账户；但此前无法回答用户计划资金是否明显超过标的流动性容量，也没有统一固定十ETF的T+0/T+1研究分类。继续只使用固定15bp/45bp成本，会让100万元和5000万元账户获得相同容量结论。
- 决策：新增`etf-capacity-impact-v1`。每个历史调仓只使用信号日及以前20根日线成交额，按买卖目标权重绝对变化估算订单金额；2% ADV为目标参与率，5%为硬上限。模型往返成本为`15 + 120 × sqrt(参与率)` bp，并与45bp压力预算比较。境内股票型ETF默认T+1，代码前缀511/513/518及明确债券、黄金、商品、海外ETF桶按T+0；未知分类失败关闭。容量结果进入`PortfolioStrategyEvidence`、DecisionHub、右侧四模块和手动账户；账户按计划总资金重新评估同一报告，不产生第二套策略。
- 已考虑的替代方案：对所有账户继续使用固定成本、把所有ETF标为T+0、或直接声称ADV足以证明可成交。固定成本不能表达规模效应；统一T+0不符合产品差异；日成交额模型没有盘口队列和部分成交信息，不能作为真实成交证明。
- 影响：100万元参考资金在2026-07-20固定十ETF复跑中最大参与率0.572%、最大模型成本24.08bp并PASS；2%目标支持资金约349.68万元，1000万元和5000万元场景超过硬参与率而FAIL。容量WATCH/FAIL/MISSING阻断新增仓位，但不能把已有持仓自动解释为必须卖出。V9最终留出仍为WATCH，不能因容量PASS升级PAPER_READY。
- 受影响的需求：TASK-027、TASK-048、US-06、US-09、FR-008、FR-009、FR-013至FR-015、FR-020至FR-022、EPV-003、EPV-006。
- 所需测试/迁移：信号时点ADV无泄漏、未来成交额扰动不变、买卖容量、缺失失败关闭、资金场景单调性、T+0/T+1分类、DecisionHub PASS/WATCH/FAIL/MISSING、账户新增/已有持仓语义和Schema往返；后续补充盘口深度、部分成交、拆单、涨跌停排队和模拟盘冲击偏差。

### ADR-054——ETF组合调仓使用权重差额会计并显式记录换手与费用
- 日期：2026-07-20
- 状态：已接受（研究执行会计修正，不改变策略参数）
- 背景：V9 回测在每个调仓日先清空全部持仓，再按目标权重买回。目标组合没有变化时仍会产生全仓换手和重复费用，低估净值并让容量观测与真实执行不一致。
- 决策：V10 保留现金和分数股研究持仓，执行日只交易当前开盘名义金额与目标权重名义金额之间的差额。每个调仓事件记录目标权重、实际权重变化、换手和费用；回测记录累计换手、平均调仓换手和累计费用。容量模型升级为`etf-capacity-impact-v2`并复用实际权重变化，分析缓存Schema升级为v5。
- 已考虑的替代方案：继续全卖全买但在文档中解释，或只修正净值而保留旧容量计算。前者保留了错误交易与费用；后者会让回测和容量使用两套交易事实。
- 影响：相同固定池和参数下，全样本净收益由V9/V1快照的+85.38%修正为V10的+97.47%，主要来自消除虚假换手和重复扣费。最后20%、25%、30%留出仍因完整滚动折不足为WATCH，不能据此升级PAPER_READY。DecisionHub和右侧模块现在可显示真实换手及费用。
- 受影响的需求：TASK-027、TASK-045、TASK-047至TASK-049、FR-008、FR-009、FR-013至FR-015、FR-020至FR-022、EPV-001至EPV-006。
- 所需测试/迁移：目标不变零换手、实际差额费用对账、事件权重一致性、容量复用实际交易差额、领域Schema往返、DecisionHub证据展示、固定十ETF多留出切片复跑和后续模拟盘成交偏差验证。

### ADR-055——拒绝双周期动量确认候选并锁定V10生产默认
- 日期：2026-07-20

### ADR-056——跳过最近一月动量候选在开发段失败后停止
- 日期：2026-07-20
- 状态：已接受（候选研究结论）
- 背景：经典股票动量研究常用 prior (2-12) return，跳过最近约一个月以减少短期反转影响。需要验证相同思想是否能改善当前固定十ETF组合，但不能用最终时间留出选择跳过窗口。

- 受影响的需求：TASK-027、TASK-045、TASK-047至TASK-051、FR-013至FR-015、FR-020至FR-022、EPV-001至EPV-006。
- 所需测试/迁移：跳过窗口端点、被排除21根K线的排名隔离、窗口约束、生产默认保护和紧凑报告配置可见；无需Schema或UI迁移。

### ADR-057——逆方差降仓候选通过研究门槛但等待模拟盘
- 日期：2026-07-20
- 状态：已接受（研究候选通过，未替换生产默认）
- 背景：V10按所选ETF的63日平均年化波动做逆波动缩放，在高波动时期仍可能保留较高动量暴露。Moreira与Muir的逆方差波动管理和Daniel与Moskowitz的动量崩溃研究提供了更强高波动降仓的候选动机，但不能直接证明中国场内ETF有效。
- 决策：增加`EtfExposureScalingModel.INVERSE_VARIANCE`显式研究开关，固定使用`(20% / selected_vol)^2`并保持25%-100%仓位边界；排名、固定池、调仓、次日开盘、成本和V10差额会计不变。候选依次通过预注册开发段和20%/25%/30%最终时间留出门槛，但Vue/Electron、DecisionHub、右侧四模块和手动账户继续默认V10`INVERSE_VOLATILITY`，直至模拟盘成交偏差和更多滚动折完成。
- 已考虑的替代方案：立即替换生产默认、继续搜索缩放指数/波动窗口/仓位下限，或只看全样本回撤升级。立即替换会绕过模拟盘；继续搜索会污染已读取的留出；只看全样本会忽略三个留出仅1/1/2折及最后20%唯一完整折超额为负。
- 影响：全样本净收益由+97.47%小幅降至+95.72%，Sharpe由1.058升至1.223，最大回撤由-21.54%改善至-13.60%；三个留出均为正收益和正超额且Sharpe高于V10，100万元容量PASS。候选状态为`RESEARCH_ONLY`，不产生新的账户或真实交易指令。
- 受影响的需求：TASK-027、TASK-045、TASK-047至TASK-052、FR-013至FR-015、FR-020至FR-022、EPV-001至EPV-006。
- 所需测试/迁移：高波动降仓、低波动无杠杆、排名一致、未来K线隔离、生产默认保护、CLI和紧凑报告证据；后续需要双轨影子净值、模拟成交偏差、部分成交和更多滚动起点，无需当前Schema或UI迁移。

### ADR-058——V13多起点影子验证通过但生产默认继续保持V10
- 日期：2026-07-20
- 状态：已接受（历史影子研究通过，未进入模拟盘或生产）
- 背景：V13已通过开发段、全样本和三个最终时间留出，但最后20%/25%/30%只有1/1/2个完整252日滚动折。需要用更多互不重叠的调仓起点判断高波动降仓改善是否只来自单一时间切片。
- 决策：按每次月度调仓执行日前一收盘到下一执行日前一收盘划分连续影子持有段，V10/V13必须使用同一证券选择、日期和会计路径。全样本51段与最后30%的19段均通过预注册门槛；最后20%/25%只作敏感性说明。结果登记为`SHADOW_RESEARCH_PASS`，但应用、DecisionHub、右侧四模块和手动账户继续默认V10；V13不得改写单标的预测或账户建议。

- 影响：全样本证券选择一致率100%，V13在13/14个V10下跌段中改善，平均亏损减少30.41%，上涨收益保留85.87%；最后30%平均亏损减少36.66%，上涨保留85.57%，最大回撤由-14.34%改善至-9.37%。结果增强了V13风险控制证据，但不能证明真实成交或未来盈利。
- 受影响的需求：TASK-027、TASK-045、TASK-047至TASK-053、FR-013至FR-015、FR-020至FR-022、EPV-001至EPV-006。
- 所需测试/迁移：连续非重叠分段、完整回测对账、证券选择一致、后续K线隔离、非暴露参数差异拒绝、CLI结构化输出；后续仍需V10/V13双轨模拟盘、理论与可得成交价格偏差、部分成交、滑点和至少一个完整市场状态周期。
